Problem → Existing Landscape

Why not existing observability tools?

Model observability

LangSmith, Langfuse, etc. Track inference latency, prompt evals, RAG quality. Oriented around model providers and pipelines — not developer workflow sessions.

Generic AI observability

Sprawls across embeddings, fine-tuning, drift. Doesn't understand sessions, tools, PR correlation, or cost-per-engineer.

Design Philosophy → Privacy

Privacy-first by default

-- Per-user visibility policy
-- Conservative defaults
CREATE TABLE visibility_policies (
  user_id                         UUID PRIMARY KEY,
  share_metadata_with_team        BOOL DEFAULT true,
  share_metadata_with_org         BOOL DEFAULT true,
  share_transcripts_with_team     BOOL DEFAULT false,
  share_transcripts_with_org      BOOL DEFAULT false
);

Transcripts are opt-in share — off by default for team and org. Metadata (cost, tool mix) is shared by default because it's aggregate-safe.

Audit log — every privileged view
actor, action, target_user, ts
alice  view_transcript  bob   09:12
alice  export_team      —     09:14

Any time a team lead or researcher views another user's session, it's recorded — actor, target, justification, IP.

Redaction before S3

Every transcript passes through packages/redaction — tokens, secrets, PII patterns stripped before the JSONL hits object storage. Raw content never written.

tech decisions ↓
Architecture → Key Tech Decisions

Why this stack?

TimescaleDB over ClickHouse
  • Same Postgres wire protocol — existing tooling, Prisma relations
  • Hypertables partition events by day automatically
  • Continuous aggregates for rollup queries at millisecond read latency
  • v1 event volume doesn't justify ClickHouse ops overhead
Hono over Express / Fastify
  • Native Bun runtime — no Node compatibility shim
  • Web-standard Request/Response — portable to Workers
  • Zod validator middleware built-in
Bun as monorepo runtime
  • Package manager + script runner + ingest server + hook compiler — one tool
  • bun build --compile produces a single-binary hook with no runtime dependency
  • Workspace catalogs enforce exact version pins across all apps
One-shot migrations runner
  • 4 services share one schema — "migrate-on-boot in each service" would race
  • Dedicated container depends on Postgres health, runs once, exits 0
  • All other services depend on service_completed_successfully

Data Model → Events Hypertable

Events firehose — partitioned by day

CREATE TABLE events (
  event_id        UUID NOT NULL,     -- UUIDv7
  session_id      UUID NOT NULL,
  user_id         UUID NOT NULL,
  ts              TIMESTAMPTZ NOT NULL,
  agent_type      TEXT DEFAULT 'CLAUDE_CODE',

  event_type      TEXT NOT NULL,
    -- SessionStart | UserPromptSubmit
    -- PreToolUse   | PostToolUse
    -- Stop         | SubagentStop

  tool_name           TEXT,
  tool_category       TEXT,
    -- fs_read|fs_write|exec|search
    -- web|task|mcp|other
  tool_duration_ms    INT,
  tool_was_denied     BOOLEAN,

  mcp_server          TEXT,
  mcp_tool            TEXT,
  subagent_type       TEXT,

  -- tokens & cost (PostToolUse / Stop)
  input_tokens        INT,
  output_tokens       INT,
  cost_usd            NUMERIC(10,6)
);
TimescaleDB hypertable
  • Auto-partitions events by ts daily
  • Continuous aggregates pre-compute per-session and per-user rollups
  • Retention policy drops raw partitions after 1 year
  • Same Postgres driver — no separate query API needed
Privacy note

tool_input_hash stores a SHA-256 of tool inputs — never the content itself. Transcript content lives in S3, never in the relational DB.

auth & batching ↓
Hook → Auth & Reliability

Short-lived tokens · local queue · resumable uploads

Auth — OIDC device flow
  • Developer runs claude-telemetry login once
  • Exchanges GitHub OAuth code for a short-lived JWT
  • JWT refreshed silently by the flusher — no re-login needed
  • No long-lived API keys stored on disk
Local queue = resilience
  • Events buffered locally if ingest is unreachable
  • Flush with backoff on network errors
  • Events are never dropped silently — surfaced in health status
Resumable transcript uploads
  • POST /v1/transcripts/{sid} accepts Content-Range
  • Large sessions (long-running compacted context) upload in chunks
  • Hook resumes interrupted uploads automatically on next Stop event
Redaction pipeline
  • Runs client-side before any bytes leave the machine
  • Strips tokens, secrets, PII patterns from transcript JSONL
  • Then zstd-compressed for upload

Dashboards → PR Intelligence

Correlating AI work to pull requests

How sessions link to PRs
  1. Hook captures git branch + git remote at SessionStart
  2. GitHub webhook fires on pull_request.opened/synchronize/closed
  3. Webhook handler matches PR branch → sessions → writes session_pr_links
  4. PR rollup computed: cost, sessions, tool mix, error rate
  5. On merge: PR bot posts summary comment to GitHub

Pinning policy
